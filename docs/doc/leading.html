The leading axis convention

Several primitive functions manipulate the right argument, or sometimes both arguments, along one or more axes. According to the leading axis model, it's best to make the primitives operate on initial axes, because the Rank modifier then allows it to apply to later axes as well. Here we'll see how this pattern works in BQN.

Other monadic functions

Not all functions work on the first axis in a straightforward manner. Transpose â‰ moves the first axis to the end, so while it focuses on the first one, it shifts every axis of ğ•©. Join âˆ¾ also works on every axis of its argument, and applies to the leading axes of ğ•©'s elements instead: these leading inner axes are matched up with the outer axes, and trailing inner axes are allowed but the elements must have rank at least as high as the argument array.

The other two monadic functions that work on high-rank arguments are Deshape (â¥Š) and First (âŠ‘). These treat ğ•© as one long list, ordered by its element indices. This ordering privileges leading axes (in fact, it is the reason for the choice of leading axes in the leading axis convention), but these functions can't really be said to work on leading axes: they apply to all axes.

The Each (Â¨) and Table (âŒœ) modifiers return functions which are the same in the monadic case. These functions simply go through all elements of the argument array without regard for its multi-dimensional structure (the operand is applied to elements in index order, matching Deshape; this matters if it has side effects). Similarly, monadic arithmetic functions do not have any sort of leading axis dependence.
